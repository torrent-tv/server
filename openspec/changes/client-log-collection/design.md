# Design: Client log collection

## Transport: batched HTTP + sendBeacon (not a WebSocket)

A live log WebSocket was considered and rejected: the sessions we most need
to debug are exactly the ones where connectivity is unstable, and a log WS
would drop together with the path it is supposed to document, adding its own
reconnect logic to maintain. Batched `POST` + `navigator.sendBeacon` on
`pagehide`:

- survives connectivity gaps (the buffer accumulates, the next flush
  delivers),
- guarantees the tail — the last seconds before a tab close or crash are the
  most valuable lines and `sendBeacon` is designed for precisely that,
- needs no connection lifecycle management at all.

Cost: not real-time. Acceptable — logs are read after the session.

## Capture: console interception, not new call sites

The codebase already routes diagnostics through prefixed console channels:
`[ice]` (webrtc-proxy.js), `[torrent-tv]` (loading.js `#logEvt`), `[hls]`
(hls-player.js, console-only by design). Intercepting `console` keeps a
single source of truth — devtools and the collector always agree — and means
zero edits at existing call sites:

- `console.error`, `console.warn`: captured entirely (unexpected failures by
  definition arrive on these).
- `console.debug`: captured only when the first argument starts with a
  known prefix (`[ice]`, `[torrent-tv]`, `[hls]`). Everything else —
  extensions, hls.js internals — is ignored.
- Global `error` / `unhandledrejection` events: captured (nothing else sees
  a compile-time import failure).

Per-segment `net-debug`-style spam stays out by not being on those channels.

## Identity and correlation

- `clientId` — `crypto.randomUUID()` per page load, generated by the logger.
- `sessionId` — the signalling server already assigns one per WebRTC session
  (`{type:"session", sessionId}`); the proxy logs it as
  `[webrtc] Session <id>`. The browser currently discards it. The logger
  records every sessionId the page receives, so one grep joins the browser
  stream to the proxy stream. Reconnects naturally appear as one clientId
  with several sessionIds.
- Entry timestamps use `performance.now()` plus the absolute epoch captured
  at logger start: monotonic within a session, mappable to server/proxy
  clock (proxy logs are UTC; observed tester offset was tz-only).

## Storage: structured stdout (decision), volume as a later upgrade

The compose stack has a single named volume (`torrent-tv-server-static`),
shared read-only with nginx for static assets — not a place to write logs.
Adding a dedicated volume means an infra change and droplet redeploy; not
justified yet. Decision: the route emits one prefixed line per entry, e.g.

    [client-log] <clientId> <sessionId?> <iso-time> <level> <message>

to stdout → `docker logs` on the droplet (`ssh do`), greppable, zero infra.
Limitation, accepted: container logs do not survive container recreation
(watchtower redeploys on release) — debug sessions are read shortly after
they happen. If retention becomes a need, add a volume + NDJSON file per day
without touching the client or the wire format.

## Abuse and failure containment

The endpoint is public and unauthenticated:

- body size limit and max entries per batch (Fastify bodyLimit + validation),
- per-IP token-bucket rate limit; over-limit batches are dropped with 429 —
  the client does not retry them (logs are best-effort, never a load source),
- message length capped per entry server-side,
- any logger failure (flush error, interception error) is swallowed:
  diagnostics must never degrade playback. The client keeps at most one
  in-flight flush; on failure entries stay in the bounded buffer (oldest
  dropped first).

## Privacy (POC posture)

Candidate lines already avoid raw addresses (`#describeCandidate` logs
`type/protocol/scope` only) — that discipline stays. Status lines may carry
torrent names; accepted for the POC with known testers, and noted as a terms
of service item for the public service (same bucket as the existing
pool-egress product risk). Enabled-by-default is likewise a POC decision to
revisit before public launch.
